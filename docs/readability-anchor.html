opus48 / readability-anchor

Claude の通知・報告文を「一読で掴める」ようにする UserPromptSubmit hook を、設計から検証まで作った記録
repo: claude-fixer  ·  plugin: opus48  ·  2026-06-17
結論。 ref-readability-essence（読みやすさ採点スキル）の visual_short アンカーを基に、Claude が返す文末の報告・通知を読みやすくするルールを毎ターン注入する hook を作った。 WHY と bad→good の few-shot を内包。A/B 検証で 4/4 シナリオ改善・平均 +70 点（baseline 22–28 → 92–95）。

02 出来たルール（注入される中身）

毎ターン、UserPromptSubmit hook が下記を additionalContext として Claude に渡す。注入文自身も「結論先出し・前置きゼロ」で書いてある（自分の原則に違反しない）。全 533 字。

WHY（なぜ効くか）

読み手の作業記憶は有限。結論を冒頭に置き一文一義にすれば、書き手が認知負荷を肩代わりでき、読み手は再読せず一読で意味を構築できる。読みやすさは理解の前提条件で、Claude が返す文末の報告・通知に最も効く。

ルール

  • do — 結論/フックを冒頭1文に（最初の改行までで主旨が分かる）
  • do — 先頭2語は内容語（挨拶・定型句で始めない）
  • do — 一文一義（主張が2つなら文を割る）
  • do — 改行は意味の単位で（結論／詳細／次アクションを分ける）
  • do — 前置きゼロ（本題から始める）
  • avoid — cushion連結語『という形で』『することができる』『に関して』『にあたって』
  • avoid — へりくだり/硬い定型句『〜を実施いたしました』『〜に寄与すると考えられます』『ご不明な点がございましたらお気軽に』
  • avoid — 二重否定・意味重複・不要語

few-shot（bad → good）

bad
ご依頼の件、修正を実施いたしました。
good
認証モジュールを修正、テストは全通過。 　login.ts の有効期限チェックを直しました。
先頭を定型句→内容語『認証モジュール』へ。へりくだりを削り結論を冒頭、詳細を別行に。
bad
エラーが発生してしまったようです。
good
ビルド失敗。原因は依存のバージョン不整合です。 　package.json を揃えれば直ります。
結論=失敗を冒頭1文に。原因と対処を改行で意味単位に分割。
bad
念のための確認となるのですが、こちらのファイルを削除する形で進めてもよろしいでしょうか。
good
このファイルを削除してよいですか。 　戻せない操作なので確認したいです。
前置きと cushion『する形で』を削り、依頼を冒頭に。理由は別行。

03 ランタイム：hook がどう効くか（シーケンス）

ユーザーが送信するたびに hook が走り、ルールが文脈に足される。Claude はそれを読んで読みやすい返信を書く。

sequenceDiagram
  autonumber
  actor U as ユーザー
  participant CC as Claude Code
  participant H as inject-readability-anchor.sh
  participant J as jq
  participant CL as Claude (opus 4.8)

  U->>CC: プロンプト送信
  CC->>H: UserPromptSubmit hook 起動
(イベントJSONをstdinへ)
  Note over H: stdin は使わず
heredoc のルールを読む
  alt jq あり
    H->>J: --arg ctx で JSON 化
    J-->>H: {hookSpecificOutput:{additionalContext:...}}
    H-->>CC: ルールJSONを stdout へ
    CC->>CL: ユーザー入力 + additionalContext(ルール)
  else jq なし
    H-->>CC: 無出力 (no-op)
    CC->>CL: ユーザー入力のみ（セッション継続）
  end
  CL-->>U: 結論先出し・前置きゼロの読みやすい返信

06 実物検証（~/playground/tmp/）

反映前に tmp でスクリプトを実行し、hook 契約を確認した。

  • OK — jq -e . で妥当な JSON。hookEventName = UserPromptSubmit、注入 533 字
  • OK — stdin に hook イベントJSONを流しても無視し、正しい additionalContext を出力
  • OK — jq が無い環境（PATH剥奪）では無出力の no-op。セッションを壊さない
RULE=$(cat <<'EOF'
【返信の読みやすさルール｜文末の報告・通知・確認依頼】
... (WHY → do/avoid → few-shot) ...
EOF
)

if command -v jq >/dev/null 2>&1; then
  jq -n --arg ctx "$RULE" \
    '{hookSpecificOutput:{hookEventName:"UserPromptSubmit",additionalContext:$ctx}}'
fi
