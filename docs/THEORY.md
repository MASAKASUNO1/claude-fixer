# readability-anchor 理論 (v0.1.2)

本ドキュメントは、UserPromptSubmit hook が毎ターン注入する「返信の読みやすさルール」(`plugins/opus48/scripts/inject-readability-anchor.sh` の heredoc)がなぜ効くのかを、認知科学の観点から人間監査向けに記述する。注入ルールは要約であり、本書がその根拠である。

---

## 1. 中心命題

**読みやすさ = 読み手の作業記憶 (Working Memory, WM) にかかる「統合コスト」の低さ。**

文章理解は逐次的なプロセスである。読み手は前から一語ずつ読みながら、意味表象を WM 内に構築・更新していく。WM は容量が小さく(Baddeley & Hitch の音韻ループ/中央実行系、Cowan の有効容量 ≒ 4±1 チャンク)、未解決の係り受け・未確定の主旨・宙づりの参照子を保持するほど負荷が上がる。

書き手が結論や要点を後置すると、読み手は「この文は何を言うのか」を仮説のまま保持し続けねばならず(= open loop)、文末まで読んで初めて遡って統合し直す(= 再読/再解析)。readability-anchor の「結論を冒頭1文」「各単位の先頭に要点」は、この open loop を最初に閉じ、後続をすべて「既に開いた枠への充填」に変える操作である。負荷のピークを書き手側へ前倒しして肩代わりする。

要するに、**読みやすい文とは、書き手が読み手の認知負荷を肩代わりするよう情報を設計した文**である。

---

## 2. 認知的根拠 — 五つのメカニズム

readability-anchor の各操作は、独立した五つの認知メカニズムに対応する。

### M1. WM 負荷の前倒し解消 (open loop を先に閉じる)

- **理論**: Just & Carpenter の reading-as-immediacy(語を読む都度ただちに統合しようとする)、Cowan の有効 WM 容量。
- **操作**: 結論先出し / 要対応の前出し / 各単位の先頭に要点。
- **効果**: 読み手は冒頭で「枠」を得るため、以降の情報を未確定のまま保持せずに済む。再読(再解析)が消える。

### M2. given-new 契約の充足

- **理論**: Clark & Haviland の given-new contract と bridging inference のコストモデル、Halliday の theme-rheme(主題-題述)構造。
- **機序**: 読み手は各文を「既知情報(given)が先、新情報(new)が後」として処理し、given を手がかりに WM/長期記憶内のアンカー(先行詞・話題)を検索してから new を接続する。アンカーが見つからない・話題が確定しないと検索が失敗し統合が遅延する(bridging コスト)。
- **操作**: 「冒頭を内容語で始める」「前置きゼロ」「改行=意味単位」。
- **効果**: 冒頭の内容語が最初のアンカーを即座に与える。挨拶・定型句で始める文は、最初の語が情報ゼロの given を空打ちさせ検索を空転させる(= demo の「薄い先頭」X-14 の正体)。意味単位の改行は「どこまでが given でどこから new か」を視覚的にマーキングし bridging を不要にする。

### M3. スキャナビリティ (二重符号化された低コスト経路)

- **理論**: Nielsen の F-pattern eye-tracking、ジャーナリズムの逆ピラミッド、Britton らの主題先行が再生成績を上げる text-structure 研究。
- **機序**: 自然読解では読み手は全文を線形に読まず、先頭・行頭を優先スキャンする。演繹順(主張→根拠)は帰納順(根拠→主張)と違い、先頭・各単位の行頭だけで主旨の骨格が復元できるため、「全文を線形処理する」高コスト経路と「先頭だけ読む」低コスト経路の両方を成立させる(冗長な二重符号化)。
- **操作**: 結論先出し / 要対応の前出し / 各単位の先頭に要点。
- **安全性との接続**: 破壊的変更が末尾に沈むと(demo の OFF L28/L37 最終行)、スキャン経路では検出不能になり、最も重要な情報が最も読まれにくい位置に置かれる(**末尾埋没 X-12**)。これは読みやすさ問題であると同時に**安全性問題**である。

### M4. 信号対雑音 — 外在的負荷の除去

- **理論**: Sweller の Cognitive Load Theory(intrinsic / extraneous / germane の三分)、Mayer のマルチメディア学習の coherence 原理・signaling 原理。
- **機序**: cushion 連結語(「という形で」「することができる」)・へりくだり定型句・過剰見出し・絵文字・二重連番は、命題内容をゼロも増やさずトークン列だけを伸ばす。読み手の WM は内容語と機能語を区別せず逐次処理するため、これらは課題本体に無関係なのに容量を食う**外在的負荷(extraneous load)**そのもの。
- **操作**: cushion 除去 / へりくだり定型句回避 / 見出しを幹だけに畳む。
- **実証**: demo で字数 775 → 453(約 41% 減)でも一次事実が全保持されたのは、削れたのが命題(intrinsic)ではなく外在的負荷だったことの直接証拠。見出しを幹だけに畳むのは視覚階層を意味階層に一致させる signaling 原理であり、過剰な構造は逆に階層の手がかりを希釈する(signaling の逆効果)。

### M5. 処理流暢性の錯覚 — 読みやすい ≠ 正しい/十分 (理論上の最重要警告)

- **理論**: Reber らの perceptual fluency-as-affect、Alter & Oppenheimer の fluency バイアス、Rozenblit & Keil の illusion of explanatory depth、Schwarz の fluency-as-truth。
- **機序**: 結論先出し・短文化は処理流暢性(processing fluency)を上げる。流暢性は主観的に心地よく、人はそれを誤帰属して「真実らしい・既知・十分・自分は理解した」と感じる。つまり**読みやすさを上げる操作は、内容が正しく十分かの判断を甘くする副作用を必ず伴う**。
- **帰結**: demo の圧縮で一次事実が保持されたのは設計と敵対検証の成果であって、流暢性向上の自動的帰結ではない。むしろ流暢な文ほど「事実が削られても気づかれにくい」。よって readability-anchor には「短くする ≠ 事実を削る」の**情報パリティ防護を理論的必須対**として組み込む。これは付加ルールではなく、流暢性錯覚への構造的対抗措置である。

## 3. 情報パリティ原則 (information-parity)

> **五つのメカニズムは「順序と圧縮」であって「削除」ではない。**

M5 の流暢性錯覚への構造的対抗として、すべての型に次の不変条件を被せる。

- **短くするのは表現だけ。一次事実は削らない。**
- **読みやすさと情報量が衝突したら、情報量を優先する。**
- 落としてよいのは非一次情報(言い換え・発見経緯・定型句・装飾・文面調整オファー)のみ。

### 一次事実(削ってはならないもの)

ファイル名 / 数値(N 件・テスト X 通過 Y 失敗)/ 失敗と flaky の区別 / 破壊的変更の有無と内容 / 残作業 / 前提・制約・条件 / 未検証・不確実の注記 / 可否(できる・できない)/ 根本原因・再現条件 / 根拠・出典 / ブロッカー。

### 出力直前の自己照合

圧縮後に1パス、敵対検証を挟む:「短くした結果、上記の一次事実が1つでも消えていないか」。消えていれば圧縮しすぎ。

### 重要な留保(検証 adv:info より)

- **hedge は content であって cushion ではない**。「おそらく」「大半のケースで」「未確認だが」「再現は不安定」などの認識的限定詞(確度・範囲・条件・未確認フラグ)は内容として保持する。flaky を「全通過」に丸めるのは事実改変であり禁止。
- **発見経緯は削ってよいが、因果・根本原因は残す**。「失敗テスト T で発覚、根本原因 R」のように経緯が一次事実(因果)を担う場合、それは load-bearing。
- **可否は一次事実**。「ロールバックできる/できない」を「することができる」の定型句として言い換え・削除すると可否情報が落ちる。
- **接続詞は論理を運ぶ**。一文一義は「独立した2主張」のときのみ適用し、1つの主張内の因果(だから)・条件(〜なら)・逆接(ただし)は接続語を保持して1文に収める。短文化は手段であって目的ではない。
- **「畳む」は記法のみ**。見出しを幹だけに畳むのは見出し記法の操作であり、畳んだ配下の一次事実は本文に残す。順序依存の手順・階層構造はフラット化しない。

## 4. 報告型タクソノミー (型ごとの適用と「削るな」リスト)

同じ三層メカニズム(WM 前倒し / given-new / スキャナビリティ)が、文・段落・文書の三層をまたいでスケールする。文レベルの「結論を冒頭」は文書レベルの「要対応を前出し」へ、「冒頭=内容語」は「各単位の先頭=要点」へ、「改行=意味単位」は「見出しを幹だけに畳む」へ対応する。

| 型 | 既定形状 | 主に効くメカニズム | 削るな(型固有の一次事実) |
|---|---|---|---|
| 完了報告 | structured(小変更は short) | 全5メカニズム | 変更ファイル名 / 数値(N件・通過X失敗Y)/ 失敗テスト名と flaky 区別 / 破壊的変更の有無と内容 / 残作業 |
| エラー報告 | short(多層原因なら structured) | M1/M2/M4 | 発生コマンド・操作 / 環境(OS/version/branch)/ エラーメッセージ本文の肝1行(原文)/ 再現性(毎回・間欠・1回) |
| 確認依頼 | short | M1/M4(+M3 で判断材料を前出し) | 影響範囲 / 不可逆性(戻せるか)/ 各選択肢の結果 / デフォルト挙動 |
| 調査まとめ | structured | M1/M3/M4/M5 | 証拠(出典・ログ・数値・再現)/ 確信度(確定・推定・仮説)/ 反証・代替仮説 / 未確認範囲 |
| 計画提示 | structured | M1/M3/M4/M5 | 各ステップの前提と依存順序 / 不可逆・破壊的ステップ / リスクとロールバック可否 / 棄却した代替案の理由 |
| 長い進捗報告 | structured | M1/M3/M4/M5 | ブロッカーと待ち相手 / 残作業と残量 / 期日影響 / 前回からの差分 |
| (全型共通) | — | 情報パリティ防護 | 上記すべて。圧縮 = 事実削除ではない |

### 形状の使い分け

- **visual_short**: 短い1件(1ファイル・テスト1件・単一エラー・単一の確認)。構造化せず2〜4行に留め、見出しを足さない。過剰分割を作らない。
- **visual_structured**: 内容量が本質的に大きい報告。要対応の前出し + 幹だけ見出し + 各単位先頭=要点。

## 5. demo 実測との接続

- **読みやすさスコア OFF 79 → ON 92(+13)**。これは「読みやすさのみ」の上昇。
- **字数 775 → 453(約 41% 減)** と **情報パリティ保持** が同時成立。
- 敵対検証で一次事実は全保持され、脱落したのは非一次情報(バグの言い換え説明・発見経緯・文面調整オファー)のみ。

この1データ点が示すのは、「負荷削減 = 内容削減」ではなく「**負荷削減 = 外在的負荷と統合コストの除去**」であること(M4 の直接証拠)。そして、一次事実保持は流暢性向上の自動的帰結ではなく設計と敵対検証の成果である(M5 の裏付け)。

## 6. 限界と適用範囲 (out of scope / failure modes)

### 6.1 対象外(別軸で担保すべきもの)

- **正しさ(factual correctness)**: 本ルールは事実の真偽・計算・コードの正当性を検証しない。読みやすい誤報は防げない。
- **情報量・網羅性(completeness)**: 情報パリティ防護が一次事実の保持を要求するのみ。必要十分な情報量の判断はしない。
- **面白さ・engagement**: 認知負荷の低さのみを扱い、魅力・説得力・語り口は評価しない。
- **tone-appropriateness(対人的な適切さ・敬意)**: へりくだり定型句の除去は読みやすさ目的であり、謝罪・悪い知らせ・交渉で必要な配慮の要否は別軸。

### 6.2 適用範囲

Claude がユーザーに返す**文末の報告・通知・確認依頼**(visual_short / visual_structured)に限る。**適用しない**もの: コード・データ・逐語引用・契約/法務文・リファレンス/仕様・API 引数表・チェックリスト・手順書、その他**情報密度自体が目的の文書**(全トークンが load-bearing で冗長削除の対象がほぼ無い)。これらは網羅性が読みやすさに優先する。

### 6.3 既知の失敗モードと防護(検証で抽出)

1. **grounds-drop(根拠が落ちる)**: 結論先出し+圧縮が根拠・出典・数値の出所を削る。→ 結論には最低1つの根拠(数値か出所)を同単位内に残す。圧縮後に敵対検証を1パス。
2. **false-certainty(不確実性の消去)**: cushion 除去が hedge まで剥がし推測を断定に格上げ。→ 認識的限定詞は content として保持。断定できない事実は断定形にしない(flaky・未再現・推測は明示)。
3. **repro-condition-drop(再現条件の脱落)**: 前提・環境・バージョン・適用範囲が「装飾的詳細」として畳まれる。→ 結果には必ず成立条件を同伴させる。
4. **verdict-forcing(演繹強制によるニュアンス潰し)**: 結論先出しは「結論が一意に定まる報告」向け。トレードオフ議論・両論併記・賛否拮抗では冒頭1文を単一 verdict でなく「論点・選択肢の枠組み」にしてよい。条件依存・探索的検討は適用対象外。
5. **density-as-goal(情報密度が目的の文書での誤適用)**: 仕様・法務・引数表では cushion 除去を機械適用すると例外条項・引数制約を削る。→ 6.2 の通り適用対象外。
6. **connective-loss(接続詞除去による論理分断)**: 「割る」のは独立2主張のときのみ。因果・条件・例外は接続語を保持し1文に収める。
7. **relational-strip(対人フレーミングの過剰除去)**: 謝罪・悪い知らせ・交渉で必要な関係性の枠組みを奪う。→ 最小限の対人フレーミングは cushion と別物として許容(順序は「要点→配慮」を維持可)。
8. **hierarchy-collapse(階層・順序情報の畳みすぎ)**: 順序依存の手順・ツリー構造をフラット化。→ 畳むのは装飾・重複・並列事実のみ。順序や包含が意味を持つ構造は番号付き/階層を保持。
9. **no-conclusion-context(結論不在文脈への強制)**: 探索中・途中経過には確定結論が存在しない。→ 冒頭1文を「現状の確度つき暫定状態(例: 原因は2候補、未切り分け)」にし、確定結論を捏造しない。
